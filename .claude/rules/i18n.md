# Rule: i18n & bilingual retrofit

Topic detail for CLAUDE.md §13 (i18n). **This file wins over a prompt.**
Egypt-based app: every user-facing string ships in **both** `src/i18n/locales/en.json`
**and** `ar.json` (Egyptian colloquial), with **full key parity**.

## Building a screen translation-ready
- Every user-facing string — labels, placeholders, **and validation/error/Alert copy** —
  goes through `t()`. No hardcoded copy.
- **A module-scope component used by a screen gets its OWN `useTranslation()`.** A row/
  card/field defined outside the screen fn can't see the screen's `t`. (This bit
  `FoodRow`, `ExtrasCard`, `DateField`, `CommentList` during the retrofit.)
- **Watch for a local `t` shadowing the hook.** The two recurring culprits:
  - an **array-destructure**: `const [p, g, t, ...] = await Promise.all([...])` — rename (`tg`).
  - a **callback param**: `onChange={(t) => …}` / `.map((t) => …)` — rename (`v`, `tag`).
- **Replace `label()` / title-case helpers with dictionary lookups.** A regex title-caser
  (`s.replace(/\b\w/g, …)`) bypasses the dictionary and has no Arabic uppercasing, so it
  silently stays English. Convert enum→label via `t(\`ns.${enumValue}\`)`, and for a
  module-scope list row pass a **`labelFor(v)` prop** from the screen.
- **An enum→dictionary lookup's keys must match the enum's VALUES verbatim.** `t(\`specialty.${s}\`)`
  resolves only if the `specialty` dict has a key for every `specialtySchema` value EXACTLY
  (`sport_performance` not `sports_performance`; include `womens_training`). A mismatch silently
  falls back to `defaultValue`/the prettifier (English) and the en/ar parity check WON'T catch it
  (both locales can be wrong-but-symmetric). After adding such a namespace, diff its keys against
  the Zod enum (`src/schemas/*`); GLM caught two missing/renamed `specialty` keys this way.
- **Not everything is translatable copy** — leave as-is: single-letter macro symbols
  (P/C/F/kcal), the `PR` badge, a `YYYY-MM-DD` format hint, and **DB-stored names**
  (food/exercise names render in their stored language; a `name_ar` column is post-pilot).
- RTL helpers live in `src/lib/rtl.ts` (`forwardChevron()`, `textStart`). Use them for
  chevrons/row alignment instead of hardcoding `chevron-forward`.

## Before you commit — run the en/ar parity check
There is **no script** in `scripts/`; `npm run test:rls` is CI-only on macOS. Run this
inline (it strips the Arabic plural suffixes `_zero/_one/_two/_few/_many/_other`, which are
legitimate extras, not mismatches):

```sh
node -e 'const en=require("./src/i18n/locales/en.json"),ar=require("./src/i18n/locales/ar.json");const f=(o,p="")=>Object.entries(o).flatMap(([k,v])=>typeof v==="object"&&v?f(v,p+k+"."):[[p+k]]);const s=k=>k.replace(/_(zero|one|two|few|many|other)$/,"");const ek=new Set(f(en).map(x=>s(x[0]))),ak=new Set(f(ar).map(x=>s(x[0])));const ma=[...ek].filter(k=>!ak.has(k)),me=[...ak].filter(k=>!ek.has(k));console.log("missing ar:",ma,"\nmissing en:",me,"\n",ma.length===0&&me.length===0?"PARITY OK":"PARITY FAIL")'
```

Also keep `tsc --noEmit` green — a bad `t()` key path doesn't fail tsc (it just renders the
key), so parity + a visual check are the real gates.

## Before claiming an i18n slice "complete" — sweep
A screen looking done is not proof. Confirm coverage with a sweep, **then** claim done
(this caught `coach/body-metric.tsx` still fully hardcoded after a premature "complete"):

```sh
for f in $(find app -name '*.tsx' -not -name '_layout.tsx'); do grep -q useTranslation "$f" || echo "$f"; done
```
The only legitimately bare result is a pure delegator with no strings (e.g.
`app/(tabs)/index.tsx`, which just renders the role-home component).

## Nav titles
The ~35 `Stack.Screen` titles in `app/_layout.tsx` are localized via the `nav` namespace
(`RootNavigator` holds `useTranslation`). They also feed the iOS back-button label. A screen
that sets its own `<Stack.Screen options.title>` overrides — the nav key is the localized
pre-mount fallback (avoids an English flash).
